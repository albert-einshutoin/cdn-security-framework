# Runbook: シークレットローテーション

> **言語:** [English](./secret-rotation.md) · 日本語

本 runbook は、フレームワークがビルド時・実行時に利用する 4 種類のシークレットのローテーション手順をまとめたものです。

| 環境変数 | 用途 | ローテーション区分 |
| --- | --- | --- |
| `JWT_SECRET` | HS256 JWT ゲート | ホット（デュアルシークレット猶予ウィンドウ） |
| `JWKS_URL` / kid | RS256 JWT ゲート | ホット（新 kid を公開 → 待機 → 旧 kid 撤去） |
| `URL_SIGNING_SECRET` | 署名付き URL ゲート | ウォーム（発行済み URL の猶予ウィンドウ） |
| `EDGE_ADMIN_TOKEN` | `static_token` ゲート | コールド（ビルド時に `dist/edge/` へ焼き付け） |
| `ORIGIN_SECRET` | オリジン認証ヘッダ | ホット（オリジン側と同時切替） |

> **大原則**：いきなり単一の新シークレットに切り替えないこと。必ず**デュアルシークレットウィンドウ**を経由し、発行中トークン、最長 TTL の署名付き URL、デプロイパイプラインの伝搬時間をすべてカバーできる期間、旧値と新値の両方を受け入れます。その後にのみ旧値を撤去します。

---

## 1. HS256 JWT シークレット（`JWT_SECRET`）のローテーション

HS256 は対称共有鍵です。発行側と検証側が揃って切替える必要があります。本フレームワークは検証側のみを持ちます（発行側は IdP や API）。

### 手順
1. **事前確認**：発行している最長の JWT 有効期限（`exp - iat`）を把握します。典型値はアクセストークン 1 時間、リフレッシュトークン 7 日間。
2. **新シークレット**を生成し、新しい名前で保存します。
   ```bash
   openssl rand -base64 32
   ```
   例：`JWT_SECRET_V2`。
3. **発行側を更新**し `V2` で署名を開始します。IdP がデュアル署名対応ならこの期間に `V1` もあわせて発行しても構いません。非対応なら、検証側のデュアル受理ウィンドウで吸収します（次ステップ）。
4. **検証側ポリシー**を両方受理に更新。
   ```yaml
   routes:
     - name: api
       auth_gate:
         type: jwt
         algorithm: HS256
         secret_envs: ["JWT_SECRET", "JWT_SECRET_V2"]
   ```
   ビルドして `dist/edge/origin-request.js` をデプロイします。
5. **待機**：`max(access_token_ttl, refresh_token_ttl) + clock_skew_sec + デプロイ伝搬時間`。7 日間のリフレッシュトークンなら 8 日を見込みます。
6. **切替**：発行側を `V2` のみに切替え、ポリシーを `V2` 検証のみに戻します。再ビルド＋再デプロイ。
7. **撤去**：シークレットストアから `JWT_SECRET` を削除します。

### 検証
- Canary：`V2` で発行したトークンで `/api/health` を叩き 200 を確認。
- ログ確認：エッジの `block_reason: "Invalid token"` が増えていなければ OK。スパイクがあれば猶予ウィンドウを延長し、まだ `V1` を撤去してはいけません。

## 2. RS256 JWKS 鍵のローテーション

RS256 は非対称鍵で、`kid` クレームと JWKS エンドポイントを軸にローテーションします。

### 手順
1. **新しい鍵**を新 `kid` で JWKS エンドポイントに公開します。旧鍵はまだ JWKS レスポンスに残します。
2. **発行側**を新 `kid` で署名する設定に更新。
3. **待機**：`firewall.jwks.cache_ttl_sec + stale_if_error_sec + 最長トークン TTL`。フレームワークは JWKS をキャッシュするため、キャッシュが更新されるまでエッジは新 `kid` を認識しません。
4. **旧鍵**を JWKS エンドポイントから削除。
5. **さらに `cache_ttl_sec` 待機**し、削除が伝搬するのを待ちます。
6. **検証**：旧鍵で署名されたトークンが `401 block_reason: "Unknown kid"` を返すことを確認します。

### ハマりどころ
- `cache_ttl_sec` が大きい（例：1 時間）と、ステップ 3 で 1 時間＋トークン TTL 待つ必要があります。TTL を短くすれば高速ローテーション可能ですが、JWKS エンドポイント負荷が増えます。
- キャッシュウィンドウを待つ前に旧 kid を削除してはいけません。旧 kid のトークンを持つユーザーがセッション中にログアウトさせられます。

---

## 3. URL 署名シークレット（`URL_SIGNING_SECRET`）のローテーション

署名付き URL は発行時点で署名が計算されるため、シークレットをローテーションするとユーザーの受信箱・メール・共有シート上の URL がすべて無効になります。

### 手順
1. **猶予ウィンドウ**を `max(signed_url.default_ttl, メール配信ウィンドウ)` で決定。72 時間が一般的な下限です。
2. **生成**：`URL_SIGNING_SECRET_V2`。
3. **検証側**を両方受理に更新。
   ```yaml
   routes:
     - name: downloads
       auth_gate:
         type: signed_url
         secret_envs: ["URL_SIGNING_SECRET", "URL_SIGNING_SECRET_V2"]
   ```
   ビルド＋デプロイ。
4. **発行側**（アプリ）を `V2` で署名する設定に更新。
5. **猶予ウィンドウ**分待機。
6. **撤去**：ポリシーとシークレットストアから `URL_SIGNING_SECRET` を削除。

### 漏洩時の強制切替
旧シークレットが漏洩した場合は猶予ウィンドウをスキップします。
1. 発行側を直ちに `V2` に切替。
2. 検証側も **V2 のみ**でデプロイ。
3. 発行済み URL は壊れる前提で運用。ユーザーには「以前のダウンロードリンクは無効です。新しいリンクをリクエストしてください」と告知。
4. 漏洩期間中に `V1` で検証されたリクエストをログ監査。

## 4. `EDGE_ADMIN_TOKEN`（static_token ゲート）のローテーション

CloudFront Functions は実行時に環境変数を読めません。`static_token` ゲートはトークンを**ビルド時**に `dist/edge/viewer-request.js` へ焼き付けるため、ローテーションは再ビルド＋再デプロイが必須です（シークレットストア更新だけでは反映されません）。

### 手順
1. **生成**：`openssl rand -hex 32`。
2. **シークレットストア**を更新（`EDGE_ADMIN_TOKEN` の値）。
3. **再ビルド**：`EDGE_ADMIN_TOKEN=<新値> npm run build`。
4. **デプロイ**：`dist/edge/viewer-request.js` を CloudFront にデプロイ。CloudFront のグローバル伝搬には 2〜5 分かかり、一部エッジは旧トークン、一部は新トークンを返す瞬間があります。管理アクセス運用はこれを見越します。
5. **周知**：管理オペレータに新トークンを共有。

### デュアルトークンウィンドウ（任意）
static_token ゲートは 1 ビルドあたり単一値です。ゼロダウンタイムにしたい場合は、同じプレフィックスに異なるトークンで複数の route を一時的に立て、切替後に統合する構成が可能です。管理パスは L7 ACL で既に絞っていることが多く、5 分の伝搬はほぼ許容されます。

## 5. `ORIGIN_SECRET`（オリジン認証カスタムヘッダ）のローテーション

オリジン認証ゲートは、オリジンリクエストに共有シークレットヘッダ（例：`X-Edge-Secret`）を追加します。ローテーションはオリジン側（ALB / NGINX / CF Worker / アプリ）で両値を受理する期間が必要です。

### 手順
1. **生成**：`ORIGIN_SECRET_V2`。
2. **オリジン**を `V1` / `V2` 両方受理に更新。
3. **エッジ再ビルド＋デプロイ**を `V2` で実施（エッジは単一値を送信するため、デュアル受理はオリジン側で持ちます）。
4. **待機**：デプロイ伝搬 ＋ 余裕（5〜15 分）。
5. **オリジン**を `V2` のみ受理に更新。
6. **撤去**：`V1` をシークレットストアから削除。

### 検証
- エッジをバイパスしてオリジンに `V1` で直接リクエスト：ステップ 5 以降は 401 を期待。
- エッジ経由リクエスト：常に 200。

---

## ローテーション後の検証（全ケース共通）

どのローテーションでも、以下を実施します。

1. **Canary**：ローテーション対象ゲートを通る合成リクエスト。
2. **ログ確認**：最初の 10 分間、対象 credential に関連する `block_reason` スパイクがないか。スパイクがあれば猶予ウィンドウを延長。
3. **ユーザー影響サンプリング**：サポートチャンネル、フロントエンドのエラーレート。
4. **監査**：旧シークレットがシークレットストア、CI 環境、キャッシュされた `.env` から完全に消えたことを確認。

---

## インシデントロールバック

ローテーションが原因でユーザー影響が出た場合：

1. **旧シークレットを復元**（オフライン保管してあったものを戻します。24 時間は即時パージしないこと）。
2. **検証側ポリシー**を両方受理に戻す。
3. **再ビルド＋再デプロイ**。
4. **原因調査**：猶予ウィンドウ不足や見落としたキャッシュが典型。

**旧シークレットはローテーション後 24 時間はオフライン vault に必ず残してください。**
